## 1. Executive Summary

The repository contains **two backends** that cannot both be the production system.
Every other problem in this codebase is downstream of that fork. Until that is resolved,
any new development risks being built on the wrong foundation.

**Top 5 critical issues (system-breaking if unfixed):**

| # | Issue | Impact |
|---|-------|--------|
| 1 | Two backends coexist (`src/backend/` FastAPI port 8000 vs `backend/` Rust port 3001) | frontend routes 404 for cascade calls because api.ts was written for FastAPI |
| 2 | `api.ts` fallback URL is `localhost:8000` — wrong when env var unset | all API calls fail without `.env` |
| 3 | `worker_dual.py` references `ModelType.PUSH_PULL` (does not exist, correct name is `PUSHPULL`) | import crash at startup |
| 4 | Redis key namespace split: code uses `retina:al:pool` / `retina:jobs:queue`, CLAUDE.md documents different keys | cross-service keys never match |
| 5 | Consumer group name mismatch: code uses `"workers"`, CLAUDE.md specifies `"retina-workers"` | worker never joins the group that backend creates |

**Handoff readiness score: 3 / 10**
The production path (Rust backend + Python worker) is architecturally coherent and ~70% implemented.
The research/experimental path (FastAPI + worker_dual.py) is 60% dead code with a broken enum.
Root directory is a session-note dump (21 markdown files) that a new developer cannot orient from.

## 6. Structural Problems

### P1 — Dual Backend (CRITICAL)

**Two complete backends exist:**

```
src/backend/app.py   FastAPI, port 8000, what README documents
backend/             Rust/Axum, port 3001, what docker-compose deploys
```

`api.ts` cascade functions call FastAPI routes. FastAPI is not in docker-compose.
Result: every cascade inference call returns a network error in the deployed stack.

The route table:

| `api.ts` calls | FastAPI has it | Rust backend has it |
|----------------|----------------|---------------------|
| `POST /api/predict/cascade` | YES (app.py) | NO |
| `GET /api/labeling/cascade/queue` | YES | NO |
| `POST /api/labeling/cascade/submit` | YES | NO |
| `POST /api/labeling/cascade/skip/{id}` | YES | NO |
| `GET /api/labeling/cascade/stats` | YES | NO |
| `GET /status` | YES | NO (has `/api/system/status`) |

**Decision required:** Does `src/backend/` remain (and get dockerized), or do the cascade
routes move to the Rust backend? The architecture in CLAUDE.md describes only the Rust stack.

### P2 — Redis Key Namespace Mismatch (CRITICAL)

| Entity | CLAUDE.md documents | Code actually uses |
|--------|---------------------|-------------------|
| Job queue stream | `retina:queue:inference` | `retina:jobs:queue` |
| Labeling pool | `retina:labeling:pool` | `retina:al:pool` |
| Consumer group | `retina-workers` | `workers` |

Both Rust (`services/redis.rs`) and Python (`redis_client.py`) use the same wrong keys,
so they are consistent with each other. **CLAUDE.md is wrong, not the code.**
The CLAUDE.md key table needs updating to match what is actually deployed.

### P3 — Stage Routing Logic in Wrong Layer

`backend/src/routes/images.rs` (lines 80–97) contains the Stage 1 vs Stage 2 routing
decision (checking label count, selecting model). CLAUDE.md §1.1 states:

> The routing rule — lives in worker.py only.

The routing logic is duplicated across two layers. The Rust backend selects the model type
before the job reaches the worker; the worker could independently disagree. This must be
resolved: either the backend dictates model type (current behavior) or the worker decides
based on `retina:system:stage` key (per CLAUDE.md §6.4). These are not reconciled.

### P4 — Stage 2 Threshold Default Mismatch

- `backend/src/config.rs` default: `100`
- CLAUDE.md §6.4: "default: 200"
- `AL_STAGE2_THRESHOLD` env var overrides both — but in a deployment without that var set,
  Stage 2 activates at 100 labels instead of the intended 200.

### P5 — `retina:system:stage` Key Never Written or Read

CLAUDE.md §6.4 describes a Stage 2 activation sequence where step 3 is:
> Set `retina:system:stage` key in Redis to `"2"`

Search result: this key is never set and never read anywhere in the codebase.
The `retina:system:stats` hash exists and is used. The stage key does not.
Stage 2 routing is derived from label count per request, not from this key.

### P6 — JWT_SECRET Reads env Directly in Route Handler

`backend/src/routes/auth.rs` line 70:
```rust
let secret = std::env::var("JWT_SECRET").unwrap_or_else(|_| "retina-dev-secret".to_string());
```

This bypasses `Config::from_env()`. The fallback `"retina-dev-secret"` will silently activate
if `JWT_SECRET` is not set, issuing tokens that are cryptographically weak.
CLAUDE.md §3.4: "JWT secret from `config.jwt_secret` — never hardcode."

### P7 — No Image Upload Endpoint

`images.rs` comments reference `POST /api/images/upload` as a prerequisite before calling
`/api/images/submit`. The upload route is not registered in the router. Clients cannot
upload files; `image_path` will always be `None`; workers will always run without pixels.

### P8 — `PipelineStage` Serialization Inconsistency

Rust:
```rust
#[serde(rename = "1")]
Unsupervised = 1,
```

Python:
```python
class PipelineStage(int, Enum):
    UNSUPERVISED = 1
```

The Rust side serializes `PipelineStage` as the string `"1"` (rename attribute).
The Python side would serialize it as the integer `1`.
When the Rust backend writes a job to Redis and Python reads it, `stage: "1"` (string)
must deserialize correctly into `PipelineStage.UNSUPERVISED = 1` (int).
Pydantic v2 coerces this for `int` enums, so it likely works in practice,
but it is fragile and should be made explicit.

## 7. Research Code Separation Problem

Research code is intermixed with production code at three levels:

**Level 1 — Root directory:** Notebooks, training scripts, and evaluation scripts sit
alongside `docker-compose.yml` and `CLAUDE.md`. A new developer cannot tell which files
are "run this" vs "this is the system."

**Level 2 — `Unsupervised_Models/` and `Supervised_Models/`:** These are complete,
standalone research implementations that are **not** wired into the production pipeline.
AdaCLIP has Decospan experiment results (v21/ directory). BGAD has a real implementation.
None of these are imported by the worker; the worker uses stubs.

**Level 3 — `src/backend/`:** The FastAPI backend contains service implementations
that overlap with the Rust backend's responsibilities. It is unclear whether `src/backend/`
is the development origin that should have been ported (and the port is incomplete)
or an experimental parallel backend.

**Required separation:**
```
research/           (was: Unsupervised_Models/, Supervised_Models/, notebooks)
  unsupervised/     AdaCLIP, PatchCore eval, PaDiM eval, WinCLIP eval
  supervised/       BGAD, Custom_Model_Push_Pull
  notebooks/        *.ipynb files
  scripts/          evaluate_model.py, fast_local_train.py, etc.
backend/            (Rust — the production backend, stays)
worker/             (Python — the production worker, stays)
frontend/           (Next.js — stays)
src/                (DELETE or rename to legacy/)
```

## 11. Issues Requiring Decision (Not Bugs — Architectural Choices)

These cannot be fixed by automation. A human must decide.

1. **Which backend is the target?** If Rust: delete `src/backend/`, rewrite cascade routes
   in Rust. If FastAPI: add Dockerfile, add to docker-compose, port security to it.

2. **Routing authority:** Does the Rust backend select the model type (current),
   or does the worker decide based on `retina:system:stage` (CLAUDE.md)?

3. **`worker_dual.py` fate:** Is the dual-model pattern (supervised + unsupervised in parallel)
   intended as the production pattern, or is `worker.py` (sequential Stage 1 → Stage 2) the target?
   Both exist; only one is wired.

4. **Stage 2 threshold:** 100 or 200? Fix config.rs and document the choice.

5. **AdaCLIP:** The research code exists with Decospan results. Is it intended to replace
   WinCLIP in the pipeline, or remain standalone research?
